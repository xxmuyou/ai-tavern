# spec-016: 本地密钥管理收敛（.env.dev 单一 SOT + sync 派生）

> **类型：** 新建  |  **依赖：** —  |  **估时：** 0.5 天  |  **状态：** ⚪ todo（详细）

---

## Context

> **配套阅读：** `.env.dev` 里 dev 阶段到底哪些必填、哪些可省（含 localhost 邮箱直登用法），见 [`docs/ops/secrets.md §1.5 Dev 阶段最小密钥集`](../ops/secrets.md#15-dev-阶段最小密钥集开发期-vs-验证期)。本 spec 只解决"密钥文件管理机制"，不涉及"代码层面哪些 key 必填"。

当前 dev 环境密钥分散在 4 处管理：

1. 根 `.env.dev` / `.env.prod` —— 历史遗留，**实际谁都不直接读它**
2. `infra/cloudflare/.dev.vars` —— Wrangler 本地读取，worker 真正用的密钥
3. `infra/cloudflare/wrangler.jsonc → vars` —— 公开变量（明文入库，前端可见 ID / URL）
4. `apps/app/.env.dev` —— 尚未创建（Expo `EXPO_PUBLIC_*` 客户端编译时注入）

**痛点：**

- 同一密钥可能要写多处（如 `STRIPE_PUBLISHABLE_KEY` 既要在 `wrangler.jsonc vars`、又要在 `apps/app/.env.dev` `EXPO_PUBLIC_STRIPE_PUBLISHABLE_KEY` 中写），改值容易漏
- `.env.dev` 的归属模糊（不被 worker / Expo 直接读），新人不知该往哪填
- 没有自动校验 / 派生机制，靠人记
- 没有 pre-commit 扫描，密钥误提交无防护

**方案讨论历史（2026-05-22）：**

评估了 1Password CLI / Infisical / Doppler 等集中式管理方案。考虑到：

- 项目当前是单人开发 + v1 阶段
- 不引入付费服务 / 外部 SaaS 依赖
- 团队化后可平滑迁移到 1Password（vault item field 引用 + `op run` 注入）

**最终采用方案：** 维持本地文件管理，但收敛到**根 `.env.dev` / `.env.prod` 作为唯一 source-of-truth**，写 `scripts/prepare-local-env.sh` 自动派生到下游位置。1Password 路径作为后续工作（v1.x 团队化时启用），见本 spec §后续工作。

v1 上线门槛 [`README.md §4`](./README.md#4-v1-上线门槛) 第 4 行明确"不允许任何 spec 以临时 hard-code / console fallback 作为 done 状态" —— 本 spec 把密钥管理从"半完成"状态推到"明确机制 + 可校验"状态。

## 目标

- 根 `.env.dev` / `.env.prod` 成为 dev / prod 密钥**唯一来源**，`.env.example` 是 schema 文档
- `pnpm local-env:dev` 自动从 `.env.dev` 派生：
  - `apps/app/.env.dev`（仅 `EXPO_PUBLIC_*` 前缀）
  - `infra/cloudflare/.dev.vars`（worker 真正消费的 secrets，按白名单）
- `pnpm dev` 启动前自动准备本地派生文件，防止漂移
- pre-commit hook（husky + gitleaks）拦截密钥误提交
- `.dev.vars` / `apps/app/.env.dev` 都加 banner `# AUTO-GENERATED by scripts/prepare-local-env.sh — DO NOT EDIT`
- prod 路径：`pnpm local-env:prod` 仅派生本地占位（`apps/app/.env.prod`，给 `expo export:web` 用），远端 Worker secrets 走 `pnpm cf:secrets:prod`

## 非目标

- ❌ 引入 1Password / Infisical / Doppler 等外部 secret 管理服务（留 v1.x 团队化）
- ❌ 双向 sync（不支持改 `.dev.vars` 反向同步回 `.env.dev`）
- ❌ 自动 `wrangler secret put` 同步 prod 远端（仅生成本地派生文件；prod 远端轮换继续手工）
- ❌ secret 加密入库（age / sops 路径）
- ❌ CI 集成（GitHub Actions secret 推送）—— v1.x 接 GHA 时做

## 实施步骤

### 1. 写 `scripts/prepare-local-env.sh`

**位置：** `scripts/prepare-local-env.sh`（bash，无外部 deps）

**CLI：**

```bash
./scripts/prepare-local-env.sh dev
./scripts/prepare-local-env.sh prod
./scripts/prepare-local-env.sh dev --dry-run
```

**行为流程：**

1. 解析 `<dev|prod>` 与 `--dry-run` 参数
2. 读取 `.env.${env}`（不存在则报错 + 提示用户参考 `.env.example`）
3. 解析为 `Record<string, string>`，忽略 `#` 开头注释行 / 空行
4. 校验：所有 key 应在 `.env.example` 中出现过（若 SOT 多出 key 不在 schema → warning，不阻断）
5. 派生：
   - **EXPO 派生**：所有 `EXPO_PUBLIC_*` key → `apps/app/.env.${env}`
   - **Worker 派生**：白名单内 key → `infra/cloudflare/.dev.vars`（仅 dev；prod 跳过 worker 派生，提示用 `wrangler secret put`）
6. 写文件时加 banner 注释，列出 sync 时间、来源
7. 输出："synced N keys → apps/app/.env.dev (M EXPO_PUBLIC_*), infra/cloudflare/.dev.vars (K worker secrets)"
8. `--dry-run` 时只打印计划，不写文件

**WORKER_KEYS 白名单（脚本顶部维护）：**

```js
const WORKER_KEYS = [
  // LLM
  'OPENAI_API_KEY', 'OPENAI_MODEL', 'DEEPSEEK_API_KEY', 'ARK_API_KEY', 'LLM_DEFAULT_ROUTE',
  // Auth
  'AUTH_TOKEN_SECRET', 'JWT_SIGNING_KEY', 'DEV_AUTH_TOKEN_TTL_SECONDS',
  'ALLOWED_ORIGINS', 'ADMIN_EMAILS', 'AUTH_SUCCESS_URL',
  // OAuth
  'GOOGLE_OAUTH_CLIENT_ID', 'GOOGLE_OAUTH_CLIENT_SECRET',
  'APPLE_SIGNIN_TEAM_ID', 'APPLE_SIGNIN_KEY_ID',
  'APPLE_SIGNIN_CLIENT_ID', 'APPLE_SIGNIN_PRIVATE_KEY',
  // Email
  'EMAIL_PROVIDER_API_KEY', 'EMAIL_FROM_ADDRESS',
  // Stripe (server-side)
  'STRIPE_SECRET_KEY', 'STRIPE_WEBHOOK_SECRET',
  'STRIPE_PRICE_PRO_MONTHLY', 'STRIPE_PRICE_CREDITS_SMALL',
  'STRIPE_PRICE_CREDITS_MEDIUM', 'STRIPE_PRICE_CREDITS_LARGE',
  // Rate / body limits
  'RATE_LIMIT_PER_MINUTE', 'LLM_RATE_LIMIT_PER_MINUTE',
  'REQUEST_BODY_LIMIT_BYTES', 'ASSET_UPLOAD_BODY_LIMIT_BYTES', 'STRIPE_WEBHOOK_BODY_LIMIT_BYTES',
];
```

> Cloudflare / AWS 部署用 key（`CLOUDFLARE_API_TOKEN` 等）**不派生**到 worker —— 它们仅给 wrangler CLI 用，从 shell env 读。

### 3. husky + gitleaks pre-commit

```bash
pnpm install   # 触发 prepare → husky 初始化 .husky/
cat > .husky/pre-commit <<'EOF'
#!/usr/bin/env sh
. "$(dirname -- "$0")/_/husky.sh"

# gitleaks: 拦截 staged 文件中的密钥
if command -v gitleaks >/dev/null 2>&1; then
  gitleaks protect --staged --redact --verbose --config=.gitleaks.toml || {
    echo "❌ gitleaks 检测到潜在密钥泄露，commit 已拦截。如确认是误报，加入 .gitleaks.toml 白名单。"
    exit 1
  }
else
  echo "⚠️  gitleaks 未安装，跳过密钥扫描。安装方式：brew install gitleaks 或 https://github.com/gitleaks/gitleaks/releases"
fi
EOF
chmod +x .husky/pre-commit
```

`.gitleaks.toml` 白名单：

```toml
title = "xtbit-apps gitleaks config"

[extend]
useDefault = true

[allowlist]
description = "Schema / placeholder files"
paths = [
  '''\.env\.example$''',
  '''docs/specs/.*\.md$''',  # spec 文档里可能有 sk_test_xxx 这种占位
  '''docs/ops/.*\.md$''',
]
```

## 验证方式

### 单元测试（脚本本身）

```bash
# .env.dev 不存在 → 应报错退出 1
mv .env.dev .env.dev.bak
./scripts/prepare-local-env.sh dev && echo "FAIL: should have errored" || echo "OK"
mv .env.dev.bak .env.dev

# --dry-run 不修改文件
ts1=$(stat -c %Y apps/app/.env.dev 2>/dev/null || echo 0)
./scripts/prepare-local-env.sh dev --dry-run
ts2=$(stat -c %Y apps/app/.env.dev 2>/dev/null || echo 0)
[ "$ts1" = "$ts2" ] && echo "OK: dry-run 未改文件" || echo "FAIL"
```

### 派生正确性

```bash
./scripts/prepare-local-env.sh dev
# 检查 apps/app/.env.dev 只含 EXPO_PUBLIC_*
grep -v '^#\|^$' apps/app/.env.dev | grep -v '^EXPO_PUBLIC_' && echo "FAIL: 非 EXPO_PUBLIC_ key 泄露" || echo "OK"

# 检查 .dev.vars 含核心 worker keys
grep -E '^(DEEPSEEK_API_KEY|STRIPE_SECRET_KEY|JWT_SIGNING_KEY)=' infra/cloudflare/.dev.vars && echo "OK" || echo "FAIL"
```

### pre-commit hook

```bash
# 故意制造泄露
echo 'STRIPE_SECRET_KEY=sk_live_FAKE_test_value_for_gitleaks' >> /tmp/fake-leak.txt
git add /tmp/fake-leak.txt
git commit -m "should fail"
# 期望：gitleaks 拦下 commit 失败
```

### 启动链路

```bash
pnpm dev
# 期望：先打印 "synced N keys"，再 concurrently 起 api + app
# worker 启动日志不应有 "missing env" 警告
```
