# 产品愿景

> 这份文档定义"我们在做什么、为谁做、为什么这样做"。具体玩法见 [`gameplay.md`](./gameplay.md)，v1 具体内容见 [`content.md`](./content.md)。

---

## 1. 产品类型

**互动故事 RPG**（Interactive Story RPG）。

不是约会模拟器，不是 AI 聊天玩具，不是恋爱手游。

核心是"故事感"与"角色塑造"：companion 不是被追求的对象，而是故事里的重要 NPC，用户与他们的关系推动情节发展。

## 2. 一句话定位

> 在虚拟现代都市里相遇 AI 角色，通过对话、事件、选择推进多维度关系，关系深度解锁新的剧情和场景 —— 没有终点，只有持续发展的故事。

## 3. 目标用户

**不限定性别，面向所有人。**

- 角色库需包含男性、女性、非二元等不同角色
- 用户自由选择想要互动的对象
- UI 与文案不默认假设用户性别（不出现"她喜欢你"这种暗示用户是男性的表达）

**目标市场：海外英文为主。**

- 产品 UI、官方角色对话、文案全部以英文为主
- 数据结构、prompt 模板、文本处理基于英文设计
- 中文等其他语言留给 v2+

## 4. 世界观

**现代都市 / 生活片。**

| 项 | 设定 |
|----|------|
| 时空 | 当代现代都市，**虚构城市**（不锁定真实地理） |
| 场景 | 咖啡馆、办公室、公寓、酒吧、街道、公园、健身房等日常都市场景 |
| 角色定位 | 同事、邻居、朋友、暧昧对象、陌生人、家人等现代都市人设 |
| 美术基调 | 动漫 / 二次元（anime / manga 风格） |

**明确不做：** 奇幻、科幻、历史、架空世界观。

## 5. 用户角色：隐形主角

用户在故事中是默认的"你"。

- **不让用户创建主角：** 不输入主角的外貌、姓名、性别、背景
- **第二人称叙述：** 所有 UI 与对话使用"you / your"
- **焦点全在 NPC：** 内容创作精力集中在 companion 身上

**为什么这样：** 降低创建门槛、最大化代入感、避免"用户想象 vs 系统呈现"的不一致。

类似 otome 经典设定 / Replika 风格。

## 6. 故事结构：开放沙盒

**没有"通关"，没有明确结局。**

- 故事是持续生成的，不是预设的章节路径
- 关系、事件、内容随时间持续涌现
- 用户可以一直玩下去，关系可以深入到任意深度
- 类似 Replika（关系陪伴） + Character.ai（多角色） + 生活模拟器（事件感）的混合

**这不意味着没有节奏感：** 系统通过事件触发、关系阈值、新角色登场制造"剧情感"，但不强加线性主线。

## 7. 核心循环

```
打开 app
  ↓
看到都市地图 / 场景列表（主界面）
  ↓
选择场景（咖啡馆 / 办公室 / 酒吧 / ...）
  ↓
进入场景，遇到一个或多个 companion
  ↓
对话（AI 生成）+ 事件 + 选择
  ↓
关系奇点维度数值变化（亲密 / 信任 / 浪漫 / 友谊 ...）
  ↓
数值变化解锁：新对话选项 / 新场景 / 新角色登场 / 特定剧情
  ↓
回到地图，重复
```

具体机制见 [`gameplay.md`](./gameplay.md)。

## 8. 角色系统：双轨

**官方精品角色（v1 提供 8-10 个）：**
- 由产品团队精心设计的角色卡（背景、人设、说话风格、关系定位）
- 体验质量基线 —— 新用户开箱即用就能获得好的内容
- 服务于产品的核心叙事调性

**用户自创角色：**
- 用户可以创建自己的角色（保留并简化现有 Guest 创建流程）
- 自由度高，个性化强
- 不进入官方推荐池（用户私人体验）

**为什么双轨：** 官方保证质量与一致性，用户创建保证灵活性与长尾。两条路径共用数据模型与对话引擎。

## 9. AI 生成边界

为了平衡**质量 × 成本 × 灵活性**，明确区分预写与 AI 实时生成：

| 内容 | 来源 |
|------|------|
| 角色卡（背景、人设、说话风格） | 预写（官方）/ 用户填写（自创） |
| 场景定义（氛围、可能事件类型） | 预写 |
| 对话内容与回应 | AI 实时生成 |
| 角色对场景 / 事件的反应 | AI 实时生成 |
| 事件细节与情节发展 | AI 实时生成（在预定义的事件类型框架内） |
| 关系数值变化规则 | 预写（规则引擎，非 AI） |

**核心原则：** AI 在被预写设定约束的框架内自由生成。预写保证基线质量与一致性，AI 保证内容的丰富与个性化。

## 10. 付费模式

**免费 + 额度限制 / 订阅去除限制。**

- **免费用户：** 每日有对话次数或 AI 调用上限
- **订阅用户：** 通过 Stripe 订阅去除限制

**v1 必须做到：**
- 免费即可完整体验产品（不是 demo wall）
- 订阅有明确的"去除限制"价值
- Stripe 集成走通

具体额度数值、订阅价格、计量方式见 [`monetization.md`](./monetization.md)。

## 11. v1 MVP 范围

| 项 | v1 必须 | v1.x 之后 |
|----|---------|-----------|
| 都市场景 | 8-10 个 | 持续增加 |
| 官方角色 | 8-10 个 | 持续增加 |
| 用户自创角色 | ✅ | 完善编辑器 |
| 关系奇点系统 | ✅ 完整 | 调优 |
| 主线预写剧情 | ❌（先做沙盒） | 可能加 1-2 条 |
| Stripe 订阅 | ✅ | 多档定价 |
| Web | ✅ | — |
| iOS | ✅ | — |
| Android | ✅ | — |
| 社区分享（用户角色互看） | ❌ | 可能加 |
| 多语言（除英文） | ❌ | 可能加中文 |

## 12. 明确不做（v1 边界）

- ❌ 主角自定义（隐形主角不变）
- ❌ 奇幻 / 科幻 / 架空世界
- ❌ 线性预设主线（开放沙盒优先）
- ❌ 社区角色分享 / 公开浏览（私密体验为先）
- ❌ 中文等多语言（英文先打磨好）
- ❌ 战斗、解谜、复杂玩法（聚焦关系与故事）
- ❌ 自动翻译用户输入

## 13. 北极星

> 用户每周打开 app 时，期待跟某个 companion 继续昨天那段对话或那个事件。

如果做到了，产品就成立了。如果做不到，再多功能都没意义。
