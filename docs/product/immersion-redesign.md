# 沉浸感与游戏性提升设计（v1.x）

> 本文档回应一个核心产品问题：**当前与 companion 的互动"很 AI、出戏、没有搭讪动力，UI 也潦草"。**
> 定位取向：**角色扮演 / 剧情冒险**（在 [`vision.md`](./vision.md) 的"AI 关系生活模拟 RPG"框架下，强化 RPG 与剧情的一侧）。
>
> 本文是**整体规划**。落地拆分见 §9，需 product owner 拍板的开放问题见 §10。
> 标注 *(暂定)* 的内容是基于现状的合理初稿，正式实施前需重新讨论。

---

## 1. 诊断：问题不是"缺系统"，是"系统没接进体验"

翻过后端代码后，一个反直觉的结论：**角色扮演引擎其实已经存在大半，只是几乎全部对玩家不可见、彼此没接线。**

已经存在但玩家感受不到的资产：

| 已有能力 | 代码位置 | 现状 |
|---|---|---|
| 7 维关系向量（closeness/trust/romance/friendship/hostility/tension/distance） | `relationships/level.ts`、`stage.ts` | 在涨，但 UI 不显示 |
| 关系阶段推导（first_contact → … → committed，含负向 hostile/estranged/strained） | `relationships/stage.ts` | 已算出 next_goal / recommended_activity，UI 不用 |
| 关系衰减 | `relationships/decay.ts` | 在跑，玩家无感 |
| **聊天每轮实时抽取情绪**（neutral/warm/playful/guarded/tense/annoyed） | `chat/signal-extract.ts` | 抽出来了，**没驱动立绘** |
| 受辱反击 / 敌意覆盖 | `chat/hostility.ts` | 已生效 |
| 事件引擎（冲突 / 开场白 / 生成） | `events/{conflict,openers,generator}.ts` | 后端有，前端入口弱 |
| 日常生活模拟（time slot / mood / availability / activity_hint / flavor_text） | `docs/product/daily-life-sim.md` + `life/` | 状态算出来了，UI 呈现弱 |
| 情绪立绘 + 3 画风 + 表情包生成管线 | spec-020 / 022 | 管线在，**聊天里立绘是死的** |
| 旁白 / 台词分离（`<narration>` 标签） | `chat/prompt.ts` | 已落地 |

**真正的三个病根：**

### 病根 A — 角色是"扁的"，且没有剧情骨架
角色卡只有 `personality / background / appearance / speech_style / relationship_role` 几段自由文本（见 `companions/index.ts`）。系统 prompt 把它们平铺进去，但**没有让角色有"当下想要什么、有什么秘密、有什么底线、主动推进什么"**。
更关键：**剧情冒险定位下没有任何编排好的故事线 / 任务钩子**。纯 sandbox 自由聊天天然没有"我想知道接下来会怎样"的拉力——这就是"像玩剧本杀但对方不是好 NPC"的根源：没有剧本，也没有动机。

### 病根 B — 所有机制对玩家不可见
关系在涨、情绪每轮在变、事件在生成、角色有今日心情和所在地——但 UI 只有"场景 + 对话框"。玩家**看不到关系进度条、看不到角色心情、不知道下一步解锁什么、不知道为什么要继续**。没有反馈回路 = 像对着空气说话 = "潦草、没动力"。

### 病根 C — 每轮抽取的情绪没驱动立绘
`signal-extract` 已经算出当轮 emotion，立绘管线也有 6 张表情图，但聊天界面的立绘静止不动。**这是代入感性价比最高的单点修复**：把这根线接上，角色当场"活"过来。

> **结论：** 提升游戏性主要是 **「接线 + 可见化 + 加剧情骨架」**，不是从零造系统。成本远低于直觉，且不浪费已有工作。

## 2. 设计原则（角色扮演 / 剧情冒险取向）

1. **角色要有自己的议程，不只回应玩家。** 好 NPC 会主动开口、追问、回避、有情绪、有底线。顺从 = 出戏。
2. **每一次互动要有"得失"。** 选择影响关系数值与剧情分支；没有得失就没有代入。
3. **进展必须可见、可期待。** 关系阶段、情绪、解锁、下一步目标——全部摆到台面上。
4. **剧情是脊柱，sandbox 是血肉。** 编排的故事节拍提供方向感，自由对话填充质感。两者都要，不能只有其一。
5. **复用 > 重造。** 优先点亮已有的 relationships / events / 情绪立绘 / daily-life 系统，而不是另起炉灶。
6. **英文优先。** 角色 persona、剧情文案、UI 全部英文（见 `vision.md §3`），中文仅团队内部文档。

## 3. 体验闭环总览

把四个支柱串成一个**有方向感的循环**，替代现在的"开对话框 → 漫无目的聊 → 关掉"：

```
进入日常主页（Today）
   → 看到角色今日状态 + 钩子台词（"她在书店，看起来心事重重"）   ← 支柱四：场景呈现/钩子
   → 被钩子驱动去搭讪 / 推进某条剧情线
   → 进入对话：立绘随情绪变化、关系条可见、当前剧情目标可见      ← 支柱三：情绪立绘；支柱二：可见化
   → 角色有议程、有秘密、推进剧情节拍，给出选择                  ← 支柱一：角色 + 剧情骨架
   → 选择改变关系数值 / 触发事件 / 解锁新立绘表情 / 推进剧情      ← 支柱二：可见反馈 + 解锁
   → 阶段跃迁 / 完成剧情节拍 → 解锁新场景、新表情、新剧情线       ← 留存钩子
   → 关系随时间衰减、角色换地点 → 制造"明天再来"的理由           ← daily-life 已有
```

## 4. 支柱一：角色塑造 + 剧情骨架

### 4.1 深化角色卡 *(暂定)*
在现有字段外，给角色补"驱动行为"的维度（新增字段或结构化进 persona）：

- **want（当下渴望）**：角色这一阶段想要什么（"想被认真对待，不想再被当备胎"）。给对话方向。
- **secret（秘密 / 软肋）**：高信任度才会揭露的东西。给"解锁"价值。
- **boundary（底线）**：什么会让 ta 变 guarded / annoyed / 疏远。给阻力。
- **flaw / contradiction（缺陷 / 矛盾）**：让角色立体、不完美。
- **opinion_of_player（对玩家的看法）**：随关系演变的一句话，喂回 prompt 制造连续感。

### 4.2 强化系统 prompt 的"NPC 性"
在 `chat/prompt.ts` 的 Rules 里加入（英文）：
- 角色**有自己的目标和情绪**，不是有求必应的助手；可以追问、调侃、回避、拒绝、设界。
- 适当**主动推进**：可以先开口、改变话题、提出一起做某事。
- 回应玩家时**带上 want / boundary / opinion_of_player 的色彩**。
- 保持"未完成感"——不要每句都给出完整答案，留钩子。

### 4.3 剧情骨架（剧情冒险定位的关键新增）
纯数值关系不够，需要**每个角色一条可推进的故事线**（character arc / quest line）：

- **arc = 有序的剧情节拍（story beats）**：每个节拍是一个有触发条件（关系阶段 / 已完成前置节拍 / 时间 / 场景）的小剧情，推进后解锁下一拍。
  - 例：`她妹妹失联` → `你陪她去找` → `发现真相` → `她向你敞开心扉`。
- **节拍可携带**：开场情境、可选的玩家分支选项、完成后的关系/解锁奖励。
- **复用已有 events 引擎**：`events/` 已有 conflict / opener / generator / resolve / templates。剧情节拍可以是"编排版事件"——把 authored beats 喂进现有事件管线，而不是新造一套。
- **authored + generated 混合**：核心节拍人工编排保证质量，过场用 LLM 生成填充。

> 这是把"剧本杀但没剧本"变成"有剧本的角色扮演"的核心。当前已拍板为轻量路线：先做每角色 1 条 3-5 拍线性故事线，不做完整多分支 arc。

### 4.4 通用 story-beat 框架（已定方向）

阶段 2 不做“某个角色的专属硬编码剧情”，而是新增一套可套娃的 companion story beat 框架：

- **Companion 是主角，scene 是舞台。** 场景入口展示“这个 companion 当前有什么事”，而不是只展示地点氛围。
- **每个 beat 是一条通用数据记录。** 它包含 stage 门槛、推荐 scene、opener、objective、奖励 key、排序；任何 official / user companion 都能挂同样结构。
- **线性优先。** v1.x 先做 3-5 拍线性 arc；分支、失败线、长期记忆改写以后再加字段。
- **体验上必须有反馈。** beat 完成后要看到阶段变化、unlock、memory、事件结果或下一目标，不能只在数据库里默默变化。
- **事件引擎是执行器之一。** authored beat 提供钩子和目标，`events` 仍负责生成互动选项与结果文案。

### 4.5 Scene story authoring 路线（已定方向）

官方 preset 内容有限，用户也需要能围绕自己喜欢的 scene 创建故事，而不是只有自由聊天。当前路线是：

- **官方内容**：内容团队维护 scene preset stories，保证质量基线。
- **用户故事**：用户在 Scene 页面选择剧情包、用户自写轻量 story，或用 AI 辅助生成 3-5 个 tasks/beats。
- **剧情包模板**：按关系定位和体验类型提供，例如 slow burn romance、healing trust、workplace tension、mystery stranger。
- **推进方式**：用户手动 `Mark as done` 完成 beat；不让系统因为发生一轮聊天就自动判断剧情结束。
- **共享方式**：scene story sharing 需按 spec-040 的 ownership 决策另行确认；默认用户私有，其他用户进度独立。
- **商业边界**：基础模板和手写免费；当前实现里 AI draft 走 Pro-only，credits 扣费需等价格决策后另接。

因此，`spec-026` 只代表基础 story beat 框架；剧情包、AI 辅助和手动完成能力由 `spec-029` 承接；Scene 页面入口、`story_id` 和 Talk/Story 边界由 `spec-040` 约束。

## 5. 支柱二：可见的关系与状态（点亮已有机制）

把后端早就算好、却藏起来的东西摆到 UI 上：

- **关系 HUD**：当前阶段（first_contact … committed）+ 阶段进度条（`stage.stage_progress` 已有）+ 当前 next_goal（已有）。
- **维度可视化** *(暂定)*：不必暴露全部 7 维原始数字（太硬核），可只显示主导维度或一个综合"亲密度"，详情页再展开。
- **即时反馈**：每轮对话后，若关系数值变化，给一个轻动效 + 提示（"信任 +2"/ "她对你冷淡了"）。这是"有得失"的可见化。
- **解锁系统**：阶段跃迁 / 完成剧情节拍 → 解锁新场景、新剧情线、新称呼、揭露秘密。给玩家**追求的目标**。
  > 已定：解锁**只靠订阅 + 关系/剧情进度**，**不消费 credits**（credits 账本仅用于自创角色生图）。
  > **2026-06-01 调整：表情立绘从「阶段解锁」改为「订阅门控的手动解锁」**——不再绑定关系阶段，Pro 用户在角色图鉴手动点击解锁并生成（免费用户跳订阅）。场景/剧情/称呼/秘密仍按关系或剧情进度解锁。
- **角色图鉴 / 回忆**：已完成的剧情节拍、解锁的立绘汇总成可回看的"回忆相册"（`vision.md` 已提"回忆"）。

## 6. 支柱三：情绪立绘接入聊天（最高性价比单点）

**这是单笔投入回报最高的改动。**

- `chat/signal-extract.ts` 每轮已产出 `Emotion`（6 种，与立绘表情一一对应）。
- 聊天界面立绘根据当轮 emotion **切换对应表情图**（资产来自 spec-020/022 的 `art_emotions`）。
- 缺失表情则回退到 neutral / `art_url`。
- 进阶：立绘切换加淡入淡出 / 轻微动效，让"她笑了""她皱眉"有视觉重量。

> 依赖：spec-020/022 的表情包生成需对该角色实际产出过表情图。对官方 seed 角色应预生成（已有 admin prewarm）。

## 7. 支柱四：场景呈现与搭讪钩子

解决"一个场景 + 对话框，没动力搭讪"：

- **角色在场景中有存在感**：立绘立在场景里，配今日心情 / 活动状态徽章（`daily-life-sim.md` 的 mood / availability / activity_hint 已有数据）。
- **钩子台词（hook line）**：每个可搭讪角色头顶一句由状态生成的预览台词 / 情境（"在读一封信，眉头紧锁" / "似乎在等什么人"）——给玩家**一个想上前的理由**。flavor_text 字段已规划。
- **聊天界面生命体征**：打字指示器（"正在输入…"）、消息分段冒出、称呼随关系阶段变化。
- **Chat Moment Image**：在有 scene context 的最新 companion 回复旁放一个小相机按钮，把最近聊天内容 + 行为/activity + 场景 + 时间 + companion + emotion 规则拼接成 prompt，生成一张完整场景瞬间图。它属于回忆/增味层，不进入 story-beat 核心推进规则。
- **微互动** *(暂定)*：送礼、轻触反应、表情回应——廉价但显著提升"她在回应我"的感觉。
- **进入动机分层**：日常钩子（今日心情）+ 剧情钩子（某条 arc 到了可推进的节拍，主页高亮提示）。

## 9. 落地拆分建议（分阶段，待拍板顺序）

按"先低成本高回报、再重投入"排：

- **阶段 0 — 立竿见影（主要前端接线）** *(建议先做)*
  - S-α：聊天界面 **emotion → 情绪立绘切换**（支柱三）
  - S-β：聊天界面 **关系 HUD + 每轮数值变化反馈**（支柱二可见化）
  - S-γ：场景界面 **角色立绘 + 今日状态徽章 + 钩子台词 + 打字指示器**（支柱四）
  > 这三个做完，"很 AI / 潦草 / 没动力"的体感问题应能改观大半，且基本不动后端。

- **阶段 1 — 角色深度**
  - S-δ：persona 字段扩展（want/secret/boundary/opinion）+ prompt Rules 强化（支柱一·角色）
  - S-ε：解锁系统（阶段 / 剧情 → 解锁立绘 / 场景 / 称呼）（支柱二·解锁）

- **阶段 2 — 剧情骨架（最大投入）**
  - S-ζ：通用 story-beat 数据模型 + 触发/完成引擎（架在 events / relationships 之上）
  - S-η：官方角色轻量 arc seed（每角色 3-5 拍）+ 主页/场景剧情钩子
  - S-θ：事件 resolve / chat / activity 完成后统一跑阶段解锁与 beat 推进，保证“100% 有反馈”

- **阶段 2b — Scene story authoring 规模化**
  - S-κ：Scene `Create story`（剧情包 / 自写 / AI 辅助 / 跳过）
  - S-λ：Scene story editor + 轻量 task editor + 手动完成 / reopen
  - S-μ：Scene story sharing / official preset governance（待 ownership 决策）

- **阶段 3 — 增味**
  - S-ι：Chat Moment Images（场景聊天瞬间图；小相机按钮 + prompt snapshot + image job）
  - 微互动（送礼 / 轻触）、回忆相册、更多事件类型

每个 S-* 落地时再各自写一份 `docs/specs/spec-0NN-*.md`。

## 10. 拍板记录与待定问题

**已拍板（2026-05-29）：**

1. ✅ **起步顺序**：**阶段 0 先行**（情绪立绘切换 + 关系 HUD/反馈 + 场景钩子，几乎纯前端接线，先验证体感提升）。
2. ✅ **剧情骨架规模**：**轻量——每角色 1 条 3-5 拍故事线**。不做完整多分支 character arc，控制内容编排成本，先跳验证。
3. ✅ **付费墙**：**解锁立绘 / 剧情节拍只靠订阅，不消费 credits**。credits 账本（spec-021）只用于自创角色的生图等，不挡剧情/立绘解锁体验。
4. ✅ **官方角色 vs 自创角色**：自建角色必须支持剧情线。路线为剧情包模板 + 用户轻量编辑 + AI 辅助草稿；sandbox 只作为跳过剧情 setup 后的兜底，不再是自建角色的唯一体验。

**仍待定：**

5. **关系数值暴露程度**：给玩家看全 7 维，还是只看阶段 + 一个综合亲密度？(硬核 vs 易上手) — 倾向后者，阶段 0 做 HUD 时定。
6. **微互动范围**：v1.x 是否纳入送礼 / 轻触，还是留到 v2？
